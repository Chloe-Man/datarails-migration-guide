📋 Before You Begin — Understanding This Process

Switching a data source in Datarails is a two-part challenge. The first part is technical: setting up the new integration so Datarails can connect to Paylocity. The second part — often overlooked — is ensuring the data inside Paylocity is clean, complete, and structured in a way that mirrors what your reports and forecasts depend on.

Simply connecting Paylocity does not mean your forecasting models will work. Field names, table structures, and data formats may differ. This guide walks you through both parts so nothing gets missed.

⚠️
Field Names vs. Field Values — Know the Difference Renaming a column is easy. Mismatched values will break your models.
+

There are two layers to field compatibility. It's important to understand which one Datarails can help with — and which one must be resolved in Paylocity before the integration goes live.

✅ Field Names — Easy Fix via Datarails Data Mapper If ADP calls a column Department and Paylocity calls it Dept, Datarails' data mapper can rename the column on ingestion. This is a simple configuration change and does not require any changes inside Paylocity.
🔴 Field Values — Must Match. Cannot Be Fixed in Datarails Alone. The actual values stored inside key fields must be consistent between ADP and Paylocity. If your Datarails models, formulas, or reports reference a specific value — like a department name — and that value changes in Paylocity, it will silently break lookups, filters, and aggregations across your entire headcount model.

Real example:

Field ADP Value (Current) Paylocity Value Impact

📌 Customer Homework Work with your Paylocity implementation team to ensure key field values — particularly Position Status, Home Department Description, and Worker Category — are configured to match your existing ADP values exactly. If Paylocity has already set different values, a full remapping and lookup reconfiguration will be required in Datarails. It is much easier to align the values in Paylocity before go-live.
STEP 03

Build the Equivalent Reports in Paylocity

Before any integration can be set up, the right reports need to exist in Paylocity first.

📄
Work with Your Paylocity Team to Create the Reports Every field identified in Step 2 must exist in a Paylocity report before the integration can begin
+

The Datarails integration with Paylocity works by reading structured report files. This means Paylocity needs to have the right reports built and configured before anything can be connected. Think of it as: if the report doesn’t exist in Paylocity, the data simply won’t be available to Datarails.

📌 What You Need to Do Share the confirmed field list from Step 2 with your Paylocity implementation team. Ask them to build the equivalent of: (1) the Employee Information Report — including all employee fields and benefits cost fields, and (2) the 401k Report — including Associate ID and contribution percentage fields. Confirm each report contains the exact fields needed before moving to Step 4.
  • Share the Step 2 field list with your Paylocity team
  • Confirm Paylocity Employee Report is built with all required fields
  • Confirm benefits cost fields are included in the employee report — or clarify if they will be a separate report
  • Confirm Paylocity 401k Report is built with Associate ID and contribution % fields
  • Spot-check a sample of employee records to confirm data looks correct and complete
STEP 04

Set Up the SFTP Integration

Once your Paylocity reports are ready, connect them to Datarails using a scheduled file transfer.

📡
What Is SFTP and How Does It Work? A plain-English explanation for finance teams
+

SFTP stands for Secure File Transfer Protocol. You don’t need to be technical to understand how it works — think of it like a scheduled file drop.

ℹ️ How It Works — Step by Step

1. Paylocity builds your report — Once your reports are configured in Paylocity (from Step 3), you set a schedule for when Paylocity should automatically generate and export an updated version of each report. For example, every night at midnight, or every Monday morning.

2. Paylocity pushes the file to a secure server — On that schedule, Paylocity automatically sends the updated report file to a secure location called an SFTP server. This is a protected file storage area — like a locked dropbox that only authorised systems can access.

3. Datarails picks it up — Datarails monitors that SFTP server. When a new file arrives, it automatically pulls the file into your Datarails file inbox and makes the updated data available to your models and reports.

The result: your Datarails headcount data updates automatically on the schedule you set, with no manual exports or uploads needed.
STEP 05

Test the Connection & Validate the Data

Work with your Datarails integration specialist to confirm the files are flowing correctly and the data looks right.

📊
Spot-Check the Incoming Data Make sure the data arriving from Paylocity looks correct before remapping
+

Before rebuilding the mapper, do a quick sense-check on the raw Paylocity data to catch any obvious issues early.

  • Total employee count in the Paylocity file matches expected headcount
  • Department values look correct and consistent (no unexpected names or blanks)
  • Position Status values are present and match the expected format
  • Associate ID column is populated for all employees with no blanks
  • Benefits cost fields are present and contain numeric values, not blanks
STEP 06

Recreate the Field Mapping in Datarails

Duplicate and update the ADP data mapper to work with Paylocity — the final step before going live.

🔄
Duplicate the ADP Mapper and Point It to Paylocity Your Datarails CSM will guide you through this — training is provided
+

In Datarails, the data mapper is what tells the system how to read and interpret the incoming file — which columns to use, what they mean, and how they connect to your model. Rather than building this from scratch, the process is to duplicate your existing ADP mapper and update it to reference the Paylocity file instead.

ℹ️ What This Involves Your Datarails CSM will provide training on how to duplicate and update a data mapper. In most cases, if the Paylocity field names and values are well-aligned with ADP (as confirmed in Step 2), this step is straightforward and quick. The mapper is updated to point to the new Paylocity file source, column names are updated where needed, and the data starts flowing into your existing models immediately.
  • Schedule a mapper training session with your Datarails CSM
  • Duplicate the existing ADP data mapper in Datarails
  • Update the data source to point to the incoming Paylocity file
  • Update any column name references that differ between ADP and Paylocity
  • Run a test extract and confirm data is populating correctly in your models
⚠️
If ADP and Paylocity Data Don’t Align — Expect Extra Work Mismatched field values add significant configuration complexity
+

If all the groundwork from Steps 1–3 was done carefully — particularly confirming that key field values like department names, status codes, and worker categories are consistent — then this final mapping step should be quick and clean.

🔴 If Field Values Don’t Match — The Complexity Goes Up If Paylocity uses different values for key fields (e.g., department names have changed, status codes are different), the mapper alone cannot fix this. Each mismatched value will require custom lookup tables or transformation logic to be built in Datarails. This adds time, requires involvement from your Datarails integration specialist, and increases the risk of errors in your forecasts during the transition. This is why aligning field values in Paylocity before go-live (Step 2) is the single most important thing you can do to keep this project on track.
✅ Best Case Scenario Field values are consistent, column names are updated in the mapper, and the Paylocity data flows cleanly into your existing headcount models with minimal rework. Your reports and forecasts continue working as before — just now powered by Paylocity.
