📋 Before You Begin — Understanding This Process

Switching a data source in Datarails is a two-part challenge. The first part is technical: setting up the new integration so Datarails can connect to Paylocity. The second part — often overlooked — is ensuring the data inside Paylocity is clean, complete, and structured in a way that mirrors what your reports and forecasts depend on.

Simply connecting Paylocity does not mean your forecasting models will work. Field names, table structures, and data formats may differ. This guide walks you through both parts so nothing gets missed.

STEP 01

Audit Your Existing ADP Data

Before touching anything, document what you have today. This becomes your reference point for everything that follows.

📂
Identify Which ADP Tables & Fields Are in Use Log every table and field currently pulled into Datarails from ADP
+

In Datarails, navigate to your existing ADP integration and review which tables are connected. For each table, document the fields that are actively referenced in your ETL steps, formulas, or reports.

📌 Customer Homework Pull a full list of ADP tables and fields currently used in Datarails. Your Datarails admin or the ETL pipeline builder should be able to export this. If you're unsure, check your existing headcount model and trace every data reference back to its source field.

Common critical fields to look for in headcount forecasting:

  • Employee ID (unique identifier)
  • Full Name / Legal Name
  • Department / Cost Center
  • Job Title / Job Code
  • Employment Type (FT, PT, Contractor)
  • FTE % (Full-Time Equivalent)
  • Hire Date
  • Termination Date
  • Location / Work Site
  • Manager / Reporting Structure
  • Compensation / Salary Band (if used in forecast)
  • Status (Active, On Leave, Terminated)
🗂️
Document Current ETL Logic & Transformations Capture how ADP data is cleaned or transformed today
+

Beyond raw fields, your Datarails pipelines may include transformation logic: renaming fields, filtering active employees, normalizing department names, or joining tables. All of this depends on ADP's specific structure.

ℹ️ Why This Matters When you switch to Paylocity, even if the data is conceptually the same, the field names and table structure may be different. Without knowing your current transformations, you won't know what to update.
  • Export or screenshot all Datarails ETL steps that reference ADP
  • Note any calculated fields or formulas that use ADP field names directly
  • Identify any filters (e.g., "Status = Active") that rely on ADP-specific values
  • List any lookup/join logic between ADP tables

⚠️
Field Names vs. Field Values — Know the Difference Renaming a column is easy. Mismatched values will break your models.
+

There are two layers to field compatibility. It's important to understand which one Datarails can help with — and which one must be resolved in Paylocity before the integration goes live.

✅ Field Names — Easy Fix via Datarails Data Mapper If ADP calls a column Department and Paylocity calls it Dept, Datarails' data mapper can rename the column on ingestion. This is a simple configuration change and does not require any changes inside Paylocity.
🔴 Field Values — Must Match. Cannot Be Fixed in Datarails Alone. The actual values stored inside key fields must be consistent between ADP and Paylocity. If your Datarails models, formulas, or reports reference a specific value — like a department name — and that value changes in Paylocity, it will silently break lookups, filters, and aggregations across your entire headcount model.

Real example:

Field ADP Value (Current) Paylocity Value Impact
Department Procurement Procurement and Sourcing Breaks lookups
Department Finance Finance and Accounting Breaks lookups
Dept → rename to Department Finance Finance Safe — values match

Key fields where value consistency is critical:

  • Department / Cost Center names — must match exactly (case-sensitive)
  • Employee Status values (e.g., "Active", "Terminated" — not "Active Employee" or "Term")
  • Employment Type codes (e.g., "FT", "PT", "Contractor" — not "Full-Time")
  • Location / Work Site names
  • Job Title / Job Code values used in forecast groupings
📌 Customer Homework Work with your Paylocity implementation team to ensure these key field values are configured to match your existing ADP values exactly. If Paylocity has already set different values, a full remapping and lookup reconfiguration will be required in Datarails — which is significant rework. It is much easier to align the values in Paylocity before go-live.
STEP 03

Confirm Paylocity Implementation Readiness

The integration can only be as good as the data behind it. Paylocity must be fully set up first.

🏗️
Paylocity Must Be Production-Ready Before Integration Do not connect Datarails to an incomplete system
+

Connecting Datarails to Paylocity is straightforward — but connecting to a Paylocity instance that is still being configured will result in incomplete, incorrect, or unstable data flowing into your forecasts.

🔴 Important Datarails can establish a data connection to any configured source. However, the connection does not guarantee the data is correct, complete, or structured the way your models expect. A successful connection ≠ a valid data migration.

Before proceeding, confirm with your Paylocity implementation team:

  • All employee records have been migrated from ADP into Paylocity
  • Historical data (past hire dates, term dates) is fully loaded — not just current employees
  • Department / Cost Center structure matches or has been remapped
  • Paylocity is live and being maintained as the system of record (not ADP)
  • A Paylocity admin has confirmed the implementation is complete
🔌
Obtain Paylocity API / Integration Credentials Datarails will need access credentials to establish the connection
+

To set up the Datarails integration, you will need credentials or connection details from the Paylocity system. Your Paylocity admin or IT team will need to provide these.

  • Confirm Paylocity API access is enabled for your account
  • Obtain Client ID and Client Secret (or equivalent credentials)
  • Confirm which data endpoints are available (e.g., employee, payroll, time)
  • Verify data permissions — ensure the integration user can access all needed tables
ℹ️ Note Your Datarails customer success team (that's us!) can guide you through the specific steps to configure the Paylocity connector once credentials are ready.
STEP 04

Reconnect the Data Source in Datarails

Set up the new Paylocity integration and rebuild your ETL pipeline.

⚙️
Add Paylocity as a New Data Source Do not delete ADP until Paylocity is fully validated
+
✅ Best Practice Keep your ADP integration active while you set up and test Paylocity in parallel. Only decommission ADP once you have confirmed that Paylocity data is complete and your models are working correctly.
  • In Datarails, add Paylocity as a new integration (do not replace ADP yet)
  • Perform an initial data sync and review the raw tables available
  • Confirm the expected tables and fields are present in the Paylocity dataset
  • Compare row counts between ADP and Paylocity for the same employee population
🔧
Update ETL Steps to Reference Paylocity Fields Remap transformations using your field mapping document from Step 2
+

Using the field mapping document you built in Step 2, go through each Datarails ETL step that references ADP fields and update them to point to their Paylocity equivalents.

  • Update all field references from ADP naming to Paylocity naming
  • Update any filter values that rely on ADP-specific value codes
  • Update any JOIN keys to use Paylocity's employee identifier
  • Re-test each ETL step individually before running the full pipeline
📌 Tip Make a copy of your existing ETL pipeline before making changes. This gives you a rollback point if something goes wrong during the transition.
STEP 05

Validate the New Data

Do not trust the new data until you have compared it against what you know to be true.

✅
Run a Side-by-Side Data Comparison ADP vs. Paylocity — same date range, same population
+

The best way to validate the new data is to run your existing reports using Paylocity data and compare the results to the last known-good output from ADP.

  • Total active headcount — does it match between ADP and Paylocity?
  • Headcount by department — do the breakdowns align?
  • New hires and terminations in a given month — do counts match?
  • Any employees in ADP that are missing from Paylocity?
  • Any unexpected employees appearing in Paylocity that weren't in ADP?
🔴 Red Flag If headcount numbers differ between the two systems, do not proceed. This indicates either incomplete data migration in Paylocity, or a structural mismatch in how active employees are defined. Escalate to the Paylocity implementation team before going live.
📊
Validate Historical Data Integrity Forecasting depends on history — make sure it transferred correctly
+

Headcount forecasting models often depend on historical trends. Confirm that Paylocity contains the historical records your models need — not just current employees.

  • Historical hire and term dates are present for past employees
  • Department history is available (if employees changed departments)
  • Data goes back far enough for your forecasting horizon (e.g., 2–3 years)
STEP 06

Rebuild or Update Existing Forecasts & Reports

Once data is validated, update your Datarails models to use Paylocity as the live source.

📈
Update Headcount Forecasting Models Reconnect forecast logic to Paylocity as the source of record
+
  • Identify all reports and dashboards that reference ADP headcount data
  • Update data source references from ADP to Paylocity in each model
  • Re-run forecasts and compare outputs to last ADP-based version
  • Get sign-off from finance stakeholder that numbers look correct
  • Archive the ADP-based version before deactivating the integration
✅ Final Step Once all reports are validated and stakeholders have signed off, you can safely deactivate the ADP integration in Datarails. Keep it inactive (not deleted) for 30 days as a safety net before fully removing it.
